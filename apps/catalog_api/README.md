# file-service

Реализация сервиса из тестового задания на Rust + Axum, гексагональная
архитектура, данные — in-memory. OpenAPI-документация отдаётся в точности
такой, какая была получена в задании.

## Архитектура

```
src/
  domain/                    — ядро, не зависит ни от Axum, ни от способа хранения
    models.rs                — сущности (FileRecord, MarkResult, RateLimitDecision)
    errors.rs                — DomainError
    ports.rs                 — трейты-порты: CandidateCatalogPort, FileStorePort, RateLimiterPort
    services/                — юзкейсы (бизнес-логика): FileNamesService, DownloadService,
                                MarkDownloadedService, AdminService

  infrastructure/            — адаптеры к внешнему миру
    config.rs                — конфигурация из переменных окружения
    persistence/              — in-memory реализация портов
      store.rs                — общее хранилище: каталог кандидата + глобальный набор файлов
      in_memory_catalog.rs     — адаптеры CandidateCatalogPort / FileStorePort поверх store.rs
      in_memory_rate_limiter.rs — адаптер RateLimiterPort с эскалацией в бан
    http/                     — Axum: хендлеры, DTO, роутинг, middleware, OpenAPI
```

Чтобы заменить in-memory хранилище на БД, нужно реализовать те же три
трейта из `domain/ports.rs` в новом адаптере и подключить его в `main.rs` —
доменный слой и HTTP-слой при этом не меняются.

## Как устроен каталог файлов

Общее количество файлов сервис не раскрывает по спецификации. Реализовано
так: при первом обращении кандидата (по `X-Candidate-Id` или по IP) ему
случайно назначается от `CATALOG_MIN_FILES` до `CATALOG_MAX_FILES` файлов
(по умолчанию 200–1000, случайно для каждого кандидата). Содержимое файлов
генерируется и кладётся в общее хранилище имя→содержимое, поэтому
`POST /api/files/download` может отдать файл по имени независимо от того,
кто его скачивает — идентификация кандидата этой ручке не нужна (как и
описано в спецификации, `x-candidate-id` у неё нет).

## Троттлинг и блокировки

Реализовано согласно тексту спецификации: превышение лимита → `429` с
`Retry-After`; продолжение попыток → `403` на 30 минут с `Retry-After`.

Алгоритм (`InMemoryRateLimiter`, в памяти, по ключу — IP-адрес клиента):

1. Фиксированное окно длиной `RATE_LIMIT_WINDOW_SECS` секунд.
2. Пока запросов в окне не больше `RATE_LIMIT_PER_WINDOW` — запрос разрешён.
3. Как только лимит превышен — `429`, и счётчик "нарушений в этом окне"
   увеличивается.
4. Если нарушений в одном окне набралось `RATE_LIMIT_BAN_THRESHOLD` —
   клиент банится на `RATE_LIMIT_BAN_DURATION_SECS` секунд, ответ `403`.
5. По истечении бана либо окна счётчики сбрасываются автоматически.

Троттлингу подчинены только `/api/files/*`. Служебные `/api/admin/*`
ручки троттлингом не ограничены (по спецификации это не требуется), но
защищены токеном (`X-Admin-Token`).

Идентификация клиента для троттлинга — всегда IP-адрес подключения,
это согласовано с админ-ручкой `DELETE /api/admin/clients/{client_ip}/throttling`,
которая тоже оперирует IP, а не `X-Candidate-Id`.

## Конфигурация (переменные окружения)

| Переменная                     | По умолчанию      | Смысл                                                        |
|---------------------------------|-------------------|---------------------------------------------------------------|
| `HOST`                          | `0.0.0.0`         | адрес, на котором слушает сервис                              |
| `PORT`                          | `8080`            | порт                                                          |
| `ADMIN_TOKEN`                   | `dev-admin-token` | токен для `X-Admin-Token`; если не задан — используется dev-дефолт, сервис явно предупреждает об этом в логах |
| `RATE_LIMIT_PER_WINDOW`         | `20`               | сколько запросов от одного клиента разрешено за одно окно     |
| `RATE_LIMIT_WINDOW_SECS`        | `60`               | длительность окна лимитера, сек.                              |
| `RATE_LIMIT_BAN_THRESHOLD`      | `3`                | после скольких превышений подряд клиент банится                |
| `RATE_LIMIT_BAN_DURATION_SECS`  | `1800` (30 мин)    | длительность бана, сек.                                        |
| `CATALOG_MIN_FILES`             | `200`              | минимальный случайный размер каталога кандидата                |
| `CATALOG_MAX_FILES`             | `1000`             | максимальный случайный размер каталога кандидата                |

`RUST_LOG` управляет уровнем логирования (например `RUST_LOG=debug`).

## Запуск

```bash
cargo run --release
```

По умолчанию поднимется на `http://0.0.0.0:8080`:

- `GET /api/files/names`
- `POST /api/files/download`
- `POST /api/files/downloaded`
- `DELETE /api/admin/candidates/{candidate_id}/progress`
- `DELETE /api/admin/clients/{client_ip}/throttling`
- `GET /openapi.json` — та самая OpenAPI-спецификация из задания, отдаётся как есть
- `GET /docs` — Swagger UI поверх `/openapi.json`

Пример быстрой проверки лимитов на маленьких значениях, чтобы не ждать:

```bash
RATE_LIMIT_PER_WINDOW=5 RATE_LIMIT_WINDOW_SECS=10 \
RATE_LIMIT_BAN_THRESHOLD=2 RATE_LIMIT_BAN_DURATION_SECS=15 \
CATALOG_MIN_FILES=5 CATALOG_MAX_FILES=8 \
cargo run
```

## Что проверено вручную

- полный цикл `names → download → downloaded` вплоть до пустого списка имён;
- `404`, когда часть запрошенных файлов отсутствует;
- `422` при превышении 3 файлов в `download` и при некорректном теле запроса;
- эскалация `429 → 403` при частых запросах и снятие бана через admin-ручку;
- `403` на admin-ручках без верного `X-Admin-Token`;
- содержимое ZIP-архива соответствует запрошенным именам.
